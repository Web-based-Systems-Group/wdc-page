Web Data Commons - Data Corpus for Large-Scale Product Matching

Anna Primpeli
Christian Bizer
Ralph Peeters

We present the WDC data corpus for large-scale product matching, which contains more than 2 million entities describing product offers. For the corpus creation we make use of the supervision found on the web in the form of schema.org offer entities annotated with schema.org identifier information. In the following sections, the necessary cleansing steps for creating the training data corpus for large-scale product matching are explained. In addition we provide the corpus for download as well as experimental baseline results.

1. Motivation for the WDC training data corpus for large-scale product matching

The task of integrating product data found on web offers has been studied as a special integration use case whose difficulty lies on the sparse and noisy nature of web data. The difficulty of the web product integration task increases as the number of data sources and product entities increases. Although there exist product data sets which can be used to evaluate product matching methods (e.g. the WDC Product Corpus and the Abt-Buy data set), they contain data that either refer to a small group of unique products or derive from a small number of sources. Supervised machine learning methods which learn sub-symbolic data representations such as deep learning, have been shown to work well on large-scale scenarios. However, such methods require large amounts of data for training. So far, such amounts of product related information have been restricted to large marketplaces such as ebay and amazon. By creating the WDC data corpus for large-scale product matching, we intend to enable the application, evaluation, and comparison of different more or less data hungry integration methods on the product domain.

Popular annotation vocabularies, like schema.org, propose a set of properties to annotate identifiers of offer entities on the web. In addition, google motivates the annotation of product identifiers with properties such as schema.org/Product/sku, gtin8, gtin13, gtin14, and mpn for more enhanced search results. The application of these annotation practices can be considered as inexpensive supervision offered in the web. We profile the adoption of identifier-related schema.org product properties using the Web Data Commons November 2017 schema.org/Product subcorpus, perform a series of cleansing and grouping steps, and create the WDC Data Corpus for Large-Scale Product Matching containing 2 million entity offers clustered in 163,650 id-groups.

3.Cleansing and Preprocessing

Considering the noisy nature of web data, we perform a series of cleansing steps. During the cleansing process we identify several false practices while annotating offer entities and develop different cleansing steps to overcome them. Below we provide details about the different false annotation practices as well as the corresponding cleansing steps and the effect they had on the resulting corpus in terms of size.

Usage of non-existing schema.org terms

A large number of offer entities are assigned wrong schema.org vocabulary terms such as schema:IndividualProduct/sku and schema:Product/Offer/sku. Despite the wrong vocabulary usage, such terms still reveal identifying information. We capture this by applying a number of regex patterns on the properties of the offer entities.

Value Normalization and filtering by value length

The next filtering step is to remove offer entities which have been annotated with invalid identifier values. As we do not possess an extensive list of all valid product identifiers, we consider the definitions of the relevant schema.org identifier properties and apply value normalization. First, all non-alphanumeric characters and initial 0s of the identifier values are removed. Considering the suggested length and structure of the identifying properties, we remove the values that after normalization are shorter than 8 characters or longer than 25 characters. Finally, we remove the values that consist only out of alphabetic characters.

Usage of multiple valid identifiers to indicate the same product

Across web sites different valid identifier values might be used to indicate the same product, e.g. one web site might add a gtin13 annotation while another one might add the mpn number to an schema.org offer entity. For capturing the different identifying values of an offer entity, we depict our data as a network graph with the distinct identifier values as vertices connected by "sameAs" edges in the case they appear in the same schema.org offer entity. The connected components of the graph form the ID-clusters of our training data corpus.

Detection and removal of taxonomy identifiers

Another common error is to annotate product taxonomy identifiers, such UNSPSC, as product identifiers. To consistently detect and remove those errors we make use of the network graph structure of our data. The taxonomy identifiers can be recognized as the nodes with a high degree and a very small clustering coefficient. The high degree reveals that the taxonomy identifier co-occurs with many other identifier values while the small clustering coefficient shows that the co-occuring identifier values do not appear in the same offer entities and thus describe different products.

Listing pages and advertisement offers detection and removal

In this cleansing step we want to remove the offers of our corpus which are part of listing pages or advertisements. We apply a heuristic for detecting listing pages and advertisements based on:

  • Amount of schema:Offer and schema:Product entities per url
  • Distribution of descriptions' length
  • Semantic connection to parent entities with properties like schema:RelatedTo and schema:SimilarTo
The applied heuristic achieves an F1 score of 94.8% (on the positive class isListingOrAd). For the evaluation we use a manually annotated set of 80 schema:Offer and schema:Product entities of our corpus, which you can download here.

Combination of information from schema:Product and schema:Offer entities

We identify that approximately 20% of the offers of our corpus have no schema.org descriptive properties. This originates in the annotation practice of attaching the identifier information to the embedded schema.org/Offer entity rather than to the parent schema.org/Product entity as proposed by google on adding structured data to product pages. An example of such an annotation practice is shown in Figure 1. In order to enrich the empty descriptive properties of the training corpus offers we additionally consider the descriptive properties of their parent items. In the example of Figure 1 we would add to the offer with id KZZ99789 the name property of its parent product entity with the value "Boys Lace Up Camo High Top Trainers". This leads to a reduction of offers with empty descriptive properties to less than 3%.

Offer Categorization

We use distant supervision from amazon product data in order to assign product category labels to the offers of our corpus. Using a publicly available amazon dataset of product reviews and metadata, we build lexica containing terms and their tf-idf scores for 26 product categories. We assign to every offer of our corpus the product category whose terms maximize the sum of the tf-idf scores of the overlapping terms. In the case that the overlap is very minimal we assign to the offer the product category "not found". For evaluating our categorization strategy, we use a manually annotated set of 985 offers which you can download here. Figure 2 shows the distribution of the different categories in our gold standard. Our categorization strategy reaches 62.3% Precision@1.

Specification Table Detection and Extraction

Based on the work of [Qiu,2015] and [Petrovski,2017] on detecting specification tables in html pages, we apply a table detection heuristic based on the following html table attributes:

  • Ratio of alphanumeric characters
  • Average number of columns per row
  • Count of table children elements
  • Count of images
  • Maximum number of columns
  • Max number of rows
  • Average length of text per row
For evaluating this heuristic we manually annotate 455 html tables found in 17 html pages with the labels "specification" and "non-specification" which you can find here. Based on our gold standard the applied heuristic reaches an F1 of 78% on the positive class which is considered the one with label "specification".

5.Corpus Files - Download

We offer the WDC training data corpus for large-scale product matching of the 2M annotated offers as json objects including the following information for every offer:

  • url: The URL where the annotated offer entity was found (considering the November 2017 Common Crawl Corpus)
  • nodeID: The subject assigned by the WDC Extractor. The combination of url and nodeID can be used as unique keys for every annotated entity.
  • clusterID: The ID of the cluster where the offer belongs.
  • identifiers: A list of all annotated identifier values of the offer entity together with their corresponding properties.
  • schema.org_description: A list of all annotated schema.org product descriptive properties such as name, description, brand etc. and their values.
  • parent_NodeID: The node ID of the parent entity of the offer.
  • relationToParent: The relation that connects the offer to its parent entity.
  • parent_schema.org_description: A list of all annotated schema.org product descriptive properties of the parent entity of the offer.

In addition we offer an English subcorpus by filtering the following public suffixes: com, net, co.uk, us, and org. The English subcorpus contains 920,885 annotated offer entities.

Finally we provide the html content of the crawled URLs containing the clustered annotated offer entities in the form of json objects with the following properties.

  • url: The URL whose content was fetched by the Common Crawl crawler (November 2017).
  • fileName: The name of the WARC file of Common Crawl that contains the related HTML response.
  • offset: The index of the HTML response in the related WARC file.
  • content: The HTML response to the call given the related URL.
