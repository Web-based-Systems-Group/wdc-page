1. Introduction

Table annotation is the task of annotating table columns and cells with semantic labels. It consists of five different tasks: column type annotation, column pair annotation, cell entity annotation, row annotation and table type detection. Public datasets are needed to evaluate table annotation approaches. The aim of this work is to create and make available datasets for benchmarking CTA and CPA models/systems using tables from the Schema.org Table Corpus.

2. Schema.org Table Corpus

The original Schema.org Table Corpus (STC) used for the creation of this benchmark was created using schema.org data from the December 2020 version of the Web Data Commons Microdata and JSON-LD corpus. The STC consists of 4.2 million relational tables covering 43 schema.org classes. Each table in the STC contains the descriptions of all entities of a specific Schema.org class that were extracted for a specific host, e.g. movie records from imdb.com or product records from ebay.com or rakuten.co.jp. All extracted entities of one Schema.org class are collected per host and subsequently passed through a pipeline consisting of three steps, (i) Attribute extraction, (ii) Removal of listing pages and sparse entities, and (iii) content-based deduplication. The result of the pipeline is the final Schema.org table for that specific host. As the Attribute extraction step is the most relevant for the creation of this benchmark, it is explained in detail in the following: First, all Schema.org attributes and their values are extracted for each of the collected entities. If an attribute contains a sub-entity instead of a literal value, these are extracted as well and presented as a list in the final table. For the attributes of sub-entities only literal values are considered. The output of this first step are tables (one per host) containing all entities of the current class and their attributes down to the second attribute level, which are the base for the creation of the CPA and CTA benchmarks. Steps (ii) and (iii) of the STC creation pipeline deal with cleansing the created tables by removing sparse entities deriving from listing pages and general deduplication of exact matching rows in the final tables. Following the cleansing steps, only attributes with a density of at least 25% are retained per table. For more information about these steps and the general STC creation process, statistics files and download links, please visit the STC website.

For the corpus only 17 more populated classes were selected. Kept only columns that have more than 50 columns in each class. The tables were extended with the subproperties of certain columns. An example is below.

(i) Language Detection: phase filtering out non-english tables and non-english rows.

(ii) Filtering out minimum of 10 rows and minimum of 3 columns

(iii) Filtering out columns that have an incorrect use of a schema.org property (checking properties under classes)

(iv) Expand columns Columns that include dictionaries(objects) with properties are expanded into new columns and old column is removed. Properties are checked to be correctly used according to schema.org. Some column such as url, image, photo and video are removed and not expanded if they are dictionaries.

3. Challenges

There are some subdatasets available for evaluating on some different challenging tasks. These are: ++ statistics (how many cols/tables for each challenge)

(i) Missing values: This subdataset includes columns from columns that have a density between 10 and 70 % of values filled. (In the CTA problem this includes xx properties and in the CPA one it includes xx properties.)

(ii) Value heterogeneity: This includes columns that fall under the same label and include values with different formats. Some examples are different date formats, different weight and width measurements. CPA Selected columns: duration, height, weight, width, date labels, price, priceRange, telephone, faxNumber, servingSize, size, salary CTA Selected columns: duration, height, weight, width, date/time labels, price, priceRange, telephone, faxNumber

(iii) Corner Cases: Corner cases are difficult cases where columns that fall under the same label are dissimilar and the ones that have different labels are similar. Inter and intra column similarity/dissimilarity included.

(iv) Random Selection: Randomly selected columns for each label of each task.
