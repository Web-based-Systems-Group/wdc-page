Statistics and Data Analysis
Dominique Ritze

Christian Bizer



This page provides basic statistics about a gold standard for mapping relational Web tables from the English WDC table corpus to DBpedia. These tables originate from the top-level-domains "com", "org", "net", "eu" and "uk". The annotations contain mappings from tables to classes, columns to properties and rows to resources.

Contents

1.Gold Standard Overview

The design goals of this gold standard were the following:

  • balanced sample wrt.
    • number of rows per table
    • topics covered
    • number of property mappings per table
  • cover a realistic subset with characteristics similar to the whole web tables corpus
  • high precision mappings (in case of ambiguous or fuzzy mappings, we decide to exclude it)

To address the requirements of having a gold standard that is as large as possible and having correspondences on instance-level (between rows in the web table and resources in DBpedia), the gold standard consists of two parts:

  • complete gold standard: 1748 tables where tables are mapped to DBpedia classes and columns to DBpedia properties
  • instance-level gold standard (subset of complete gold standard): 233 tables, where rows are mapped to DBpedia resources

The selection strategy for integrating tables in the gold standard was to first take a random sample from the complete corpus. The majority of tables from this sample is non-relational (i.e. layout tables) or cannot be mapped to DBpedia. However, these tables can be helpful to distinguish between tables where we can find DBpedia resources and tables where an algorithm should detect that no resources can be mapped. To increase the amount of actually mappable tables in the gold standard, we also specifically searched for tables that potentially map (i.e. have at least some overlapping values). For this task we used the Mannheim Search Joins Engine with input tables from DBpedia covering different topics.

With the largest amount of tables having less than 20 columns and no mapping to a DBpedia property at all, we address the representativeness requirement. Thus, we try to cover a realistic scenario with a lot of non-matchable tables with only few rows. Dividing the gold standard according to the two dimensions can also help to see for which tables a system performs well and where it can be improved.

Concerning the mapping task, we call a table a content table if it contains relational data and has at least one column (i.e. the "key column") that is mappable to DBpedia. In the complete gold standard, we have about 763 of these content tables. They are usually larger than the other tables (Ø 206.87 rows, Ø 4.7 columns). Since it is not useful to assign DBpedia classes or properties to non-content tables, only content tables are annotated with this information.

Altogether, the gold standard contains mappings to 91 different DBpedia classes which ensures a broad coverage of different topics. We grouped these classes based on their super classes (called categories) and show their distribution in Figure 1. All classes which only have the super class "Thing" are assigned to this category. One example is the class "Drug". Other classes can be consolidated to a category covering different but related classes, e.g. the category "Organization" contains tables about companies, universities political parties, airlines, schools etc.

3. Column Characteristics

In the complete gold standard all tables together contain 7983 columns of which 4100 columns originate from content tables. Since it is again useless to map a column in a layout table to a DBproperty, we only create property matchings for column of content tables. This results in 2084 mappings to DBpedia properties. 750 of these are mappings from the key column to rdfs:label, the other ones are from non-key columns to properties of the DBpedia ontology namespace. All the mappings together cover 298 different properties.

Figure 2 shows the top 20 properties (without rdfs:label).

Fig. 2 - Top 20 DBpedia properties

Only having columns of a certain data type could also bias the gold standard and encourage system that focus only on matching this data type. Thus, we took almost the same column data type distribution as it is present in the corpus. While the gold standard covers 65% columns of type string, the whole corpus has 68%, similar holds for the other data types.

Including columns with different data types shows the strengths and weaknesses of systems/algorithms according to the various similarity functions.

4. Instance Gold Standard

The instance-level gold standard contains 233 tables that have been annotated with a DBpedia class, DBpedia properties and DBpedia resources. Again, we used the Mannheim Search Joins Engine to obtain tables that are likely to be content tables. This subset of the complete gold standard covers 33 DBpedia class and 26,124 instance correspondences. To get an impression about the topics, we used the same categories as before.

Figure 4 shows the distribution of tables per categories.

Fig. 4 - Distribution of Tables per Category

The distribution of tables according to the categories does not seem to be very balanced. This results from the fact that tables of different categories significantly differ in their amount of instance correspondences. Having a look at the amount of correspondence per category (Figure 5), the difference becomes visible.

While tables of the category "Species" can be mapped to a lot of resources, tables about "Architectural Structure" have a tendency to contain less correspondences. Similar holds for the number of mappings to properties, e.g. tables of the category "Populated Place" can on average be mapped to 4 properties while tables of the category "Person" only to 1.5 properties. Besides other characteristics, these differences significantly influence the simplicity of matching a certain table. Having a lot of property mappings can help to find better correspondences for entities in the table since more information is available. The same holds vice versa, with more correspondences to resources, the mappings of properties could be improved.
