Statistics and Data Analysis

This page provides basic statistics about a gold standard we created for the relational Web tables in the English WDC table corpus that originate from the top-level-domains "com", "org", "net", "eu" and "uk". The web tables have been annotated with DBpedia classes, properties and resources.

Contents

1.Gold Standard Overview

We developed the gold standard with several goals in mind:

  • balance
    • number of rows
    • topics
    • number of property mappings
  • cover a realistic subset with similar characteristics than the whole web tables corpus
  • high precision mappings

Since on the one hand we like to have a large gold standard covering a certain amount of realisitc web tables but on the other hand to have a gold standard to test instance matching between entities in the web table and resources in DBpedia, our gold standard is divided into 2 parts:

  • large gold standard: 1748 tables where tables are assigned to DBpedia classes and columns to DBpedia properties
  • instance gold standard (also included in the large gold standard): 233 tables where additionally rows are annotated with DBpedia resources

For the large gold standard, we took a certain amount of random web tables from the English WDC table corpus such that it represents the whole corpus. It contains not only relational tables but also layout tables or tables that cannot be mapped to DBpedia at all. However, these tables can be helpful to distinguish between tables where we can find DBpedia resources and tables where an algorithm should detect that no resources can be mapped. Due to the large amount of tables that cannot be mapped to DBpedia, we decided afterwards to specifially search for tables that potentially map. Therefore, we used the Mannheim Searchs Join Engine with input tables from different topics. This way, it is more probable to get web tables which really contain content.

Having the largest amount of tables that have less than 20 columns and no mapping to a DBpedia property at all, is representative for the corpus. Thus, we try to cover a realistic scenario with a lot of non-matchable tables with only few rows. Dividing the gold standard according to the two dimensions can also help to see for which tables a system performs well and where it can be improved. We call tables content tables if they are not layout tables and thus might contain information that is also available in DBpedia. In the large gold stadnard, we have about 763 of these content tables. These are usually larger than the layout tables (Ø 206.87 rows, Ø 4.7 columns). Since it is not useful to assign DBpedia classes or properties to layout tablea, we only labelled content tables. Altogether, we assigned 91 different DBpedia classes which should ensure the covering of different topics. We summarized the classes based on their super classes (called categories) to get an impression of the most frequent categories. Figure 1 shows the distribution on the categories.

Other classes can be consolidated to a category covering different but related classes, e.g. the category "Organization" contains tables about companies, universities political parties, airlines, schools etc.

3. Column Characteristics

In the large corpus all tables together contain 7983 columns of which 4100 columns origin from content tables. Since it is again useless to map a column in a layout table to a DBproperty, we only create property matchings for column of content tables. This results in 2084 mappings to DBpedia properties. 750 of these are mappings from the key column to rdfs:label, the other ones are from non-key columns to properties of the ontology namespace. All the mappinsg cover 298 different porperties.

Figure 2 shows the top 20 properties (without rdfs:label).

Fig. 2 - Top 20 DBpedia properties

Only having columns of a certain data type could also bias the gold standard and encourage system that focus only on matching this data type. Thus, we took almost the same column data type distribution as it is present in the corpus. While the gold standard covers 65% columns of type string, the whole corpus has 68%, similar holds for the other data types.

Figure 3 shows the distribution of column data types.

Including columns with different data types shows the strenghts and weaknesses of systems/algorithms according to the various similarity functions.

4. Instance Gold Standard

Our smaller corpus, Instance Gold Standard, contains 233 tables that have been annotated with a DBpedia class, DBpedia properties and DBpedia resources. Again, we used the Mannheim Search Joins Engine to obtain tables that are likely to be content tables. This subset of the gold standard covers 33 DBpedia class and 26124 instance correspondences. To get an impression about the ropics, we used the same categories as before.

Figure 4 shows the distribution of tables per categories.

Fig. 4 - Distribution of Tables per Category

The distribution of tables according to the categories does not seem to be very balanced. This results from the fact that tables of different categories significantly differ in their amount of instance correspondences. Having a look at the amount of correspondence per category (Figure 5), the difference becomes visible.

Fig. 5- Distribution of the Correspondences per Category.

However, since we started to annotate tables covering the topics "Populated Place" and "Work", the gold standard still contains more tables and correspondences of this category. Especially for the instance gold standard, including various classes plays an important role since they show different matching characteristics.

While tables of the category "Species" can be mapped to a lot of resources, tables about "Architectural Strucure" contain tendencially less correspondences. Similar holds for the number of mappings to properties, e.g. tables of the category "Populated Place" can on average be mapped to 4 properties while tables of the category "Person" only to 1.5 properties. Besides other characteristics, these differences significantly influence the simplicity of matching a certain table. Having a lot of porperty mappings can help to find better correspondences for entities in the table since more information is available. The same holds vice versa, with more correspondences to resources, the mappings of properties could be improved.
